## 🔍 详细对比

### 1. ConnersHua RuleGo + TG-Twilight AWAvenue-Ads-Rule

**当前实现** (`build-reject-domainset.ts`)：

```typescript
// 简单处理，直接按行解析
for (const line of lines) {
  const parts = line.split(',');
  const ruleType = parts[0];
  const value = parts[1];

  if (ruleType === 'DOMAIN' || ruleType === 'DOMAIN-SUFFIX') {
    // 仅验证 TLD
    const validation = validator.validate(value, { source: RuleSource.RemoteList });
    if (!validation.valid) {
      stats.invalidTlds++;
      continue;
    }
    domainTrie.add(value, ruleType === 'DOMAIN-SUFFIX');
  }
}
```

**理想流程** (surge-master-2 中的 RulesetOutput)：

- ❌ 不做 tldts 规范化（保持原始格式）
- ✅ 基础 Trie 去重
- ❌ 不做活性检测
- ✅ 基础格式验证（哈希检测等）

**差异**：当前实现缺少完整的规则解析，没有处理 IP-CIDR、DOMAIN-KEYWORD 等其他规则类型。

## 🎯 建议改进

- 保留原始规则特性（如通配符、关键词）

- 保持规则的原始意图

## 📚 规范化函数说明

### 不同级别的规范化函数

```typescript
// 1. 不做规范化
// 直接使用原始域名，仅做基本验证

// 2. 轻量级规范化 (fastNormalizeDomainWithoutWww)
// - 去除 www 前缀
// - 去除 xml-v4 前缀
// - 处理 www. 开头的子域名
// - 过滤 IP 地址
// - 验证 TLD 合法性（isIcann || isPrivate）

// 3. 完整规范化 (fastNormalizeDomain)
// - 过滤 IP 地址
// - 验证 TLD 合法性
// - 保留完整的子域名结构

// 4. 宽松规范化 (looseTldtsOpt)
// - 允许私有域名
// - 允许非标准 TLD
// - 用于处理特殊规则（如通配符）
```

### 规范化级别选择原则

1. **Surge 规则文件**（ConnersHua、TG-Twilight）

   - 已经是标准化格式
   - 仅需 TLD 验证，不做域名变换

2. **HOSTS 文件**

   - 需要轻量级规范化
   - 去除常见前缀（www、xml-v4）

3. **纯域名列表**

   - 需要完整规范化
   - 严格的格式验证

4. **AdGuard 过滤器**
   - 需要宽松规范化
   - 支持复杂的规则语法
