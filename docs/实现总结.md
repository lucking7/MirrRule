# TLD 验证和规则优化实现总结

## 1. 完善 TLD 验证

### 分级验证策略

根据 Surge-master-2 的逻辑，实现了基于数据源的分级 TLD 验证：

#### 数据源类型和验证策略

| 数据源类型      | 验证策略 | 说明                      |
| --------------- | -------- | ------------------------- |
| 本地规则文件    | 宽松验证 | 接受 ICANN TLD 和私有后缀 |
| 远程 Hosts 文件 | 宽松验证 | 如 StevenBlack hosts      |
| 远程域名列表    | 宽松验证 | 如 AdGuard CNAME trackers |
| AdGuard 过滤器  | 严格验证 | 只接受 ICANN TLD          |

### 关键文件

1. **EnhancedTldValidator** (`lib/enhanced-tld-validator.ts`)

   - 实现了分级验证逻辑
   - 支持 ICP TLD 白名单（130 个中国域名 TLD）
   - 支持特殊用途 TLD（localhost、local 等）

2. **AdGuard 过滤器解析** (`lib/parse-filter/adguard-filter.ts`)

   - 解析 AdGuard 格式的过滤规则
   - 提取域名和修饰符

3. **验证脚本** (`scripts/check-reject-tld.ts`)
   - 检测 reject 规则中的非法 TLD
   - 支持 `--fix` 参数自动修复

### `adtago.s3.amazonaws.com` 分析结果

- **不是非法 TLD**：它使用的是私有后缀 `s3.amazonaws.com`
- **处理策略**：
  - 在 AdGuard 过滤器中会被过滤（严格模式）
  - 在本地规则文件中会被保留（宽松模式）
- **结论**：作为追踪器域名，应该保留在 reject 规则集中

## 2. Reject 规则合并处理

### RejectRuleMerger (`sync/rule-reject-merger.ts`)

实现了智能的 reject 规则合并，包括：

1. **域名优化**

   - 使用 HostnameTrie 自动合并父子域名
   - 例如：`.example.com` 会包含所有 `*.example.com`

2. **IP 段优化**

   - 使用 fast-cidr-tools 合并相邻 IP 段
   - 支持 IPv4 和 IPv6

3. **白名单处理**

   - 参考 Surge-master-2 的预定义白名单
   - 包括崩溃报告服务、本地域名、重要服务等

4. **TLD 验证**
   - 对每个域名进行 TLD 合法性检查
   - 本地文件使用宽松验证策略

### 集成到同步流程

在 `module-sync.ts` 中集成了 reject 规则合并：

- 在同步模块之前执行合并
- 备份原文件到 `block.list.bak`
- 直接覆盖原文件 `block.list`
- 自动提交到 Git 仓库

## 3. 规则优化功能

### 新建独立工作流 (`.github/workflows/optimize-rules.yml`) [已集成到 main.yml]

**更新：优化功能已经集成到 main.yml 的 validate 步骤之前，不再需要独立的工作流**

- cdn.list, stream.list, microsoft.list, lan.list, apple.list

## 4. 命令行优化工具

### RuleOptimizer (`sync/rule-optimizer.ts`)

创建了功能强大的命令行优化工具：

#### 特性

1. **灵活的文件选择**

   - `--include`：仅优化指定文件
   - `--exclude`：额外排除文件
   - `--exclude-only`：替换默认排除列表

2. **优化模式**

   - `--ip-only`：仅优化 IP 规则
   - `--domain-only`：仅优化域名规则
   - `--dry-run`：预览模式，不修改文件

3. **详细报告**
   - 按类型分组（域名、IP、混合）
   - 显示优化前后的规则数量
   - 计算压缩率

#### 使用示例

```bash
# 优化所有文件（排除默认列表）
npm run optimize:rules

# 预览模式
npm run optimize:rules:dry

# 仅优化指定文件
npx tsx rule-optimizer.ts --include facebook.list,twitter.list

# 额外排除文件
npx tsx rule-optimizer.ts --exclude custom.list

# 查看帮助
npm run optimize:rules:help
```

## 5. 与 Surge-master-2 的对比

### Surge-master-2 的策略

1. **自定义构建脚本**

   - 使用 `# $ custom_build_script` 标记跳过自动处理
   - reject.conf、domestic.conf 等文件有专门的构建逻辑

2. **自动优化**
   - 对未标记的文件自动应用 IP 段合并和域名优化
   - 在 `FileOutput` 类的 `writeToStrategies` 方法中实现

### 我们的实现

1. **更灵活的排除机制**

   - 支持命令行参数动态指定
   - 可以完全自定义排除列表

2. **独立的优化工具**

   - 可以单独运行，不依赖构建流程
   - 支持预览模式和详细报告

3. **工作流分离**
   - 优化逻辑独立于主构建流程
   - 更容易维护和调试

## 注意事项

1. 本地文件中的"非法 TLD"会被保留（宽松验证）
2. AdGuard 过滤器中的私有后缀会被过滤（严格验证）
3. 优化是破坏性操作，会改变原始文件结构
4. 白名单包含重要的崩溃报告和分析服务
5. 默认排除的文件是基于用户需求和 Surge-master-2 的实践
